## 一、系统概述

### 1.1 设计目标

探索系统是一个替代原有"调查"行动的全新模块，旨在提供：
- 更丰富的叙事体验
- 事件碎片收集玩法
- 多分支决策影响结局
- 能力值影响调查深度

### 1.2 核心流程

```
探索行动 → 获得事件碎片 → 事件背包检测 → 完整事件触发 → 事件内决策 → 结局
```

## 二、系统架构

### 2.1 核心数据结构

```python
class EventFragment:
    """事件碎片"""
    fragment_id: str           # 碎片唯一ID，如 "mystery_club_1"
    event_id: str              # 所属事件ID，如 "mystery_club"
    name: str                  # 碎片名称，如 "神秘的社团传单"
    description: str           # 碎片描述
    required_abilities: dict   # 调查此碎片需要的能力值
                              # {"说服": 5, "田野调查": 3}
    ability_bonus: dict        # 拥有此碎片后提供的能力加成
                              # {"神话学": 2}

class DecisionNode:
    """决策节点"""
    node_id: str               # 节点ID
    description: str            # 当前情况描述
    choices: list              # 选项列表
    # 每个choice包含：
    # - text: 选项文本
    # - ability_check: 能力检定 {"技能": 需要值}
    # - success_outcome: 成功结果
    # - failure_outcome: 失败结果
    # - next_node: 下一节点ID（成功/失败分别）

class StoryEvent:
    """完整事件"""
    event_id: str              # 事件唯一ID
    title: str                 # 事件标题
    intro: str                 # 事件介绍文本
    fragments_needed: list     # 触发此事件需要的碎片ID列表
    decisions: dict            # 决策节点 {node_id: DecisionNode}
    start_node: str            # 起始节点ID
    possible_outcomes: list    # 可能的结局列表
    # 结局包含：
    # - outcome_id: 结局ID
    # - title: 结局标题
    # - description: 结局描述
    # - effects: 结局效果 {sanity: -10, knowledge: 5}
    timeline: str              # 完整时间线（仅供开发者阅读的注释）

class EventBackpack:
    """事件背包"""
    fragments: list            # 已收集的碎片列表 [fragment_id, ...]
    completed_events: list      # 已完成的事件ID列表

    def can_trigger(self, event: StoryEvent) -> bool:
        """检查是否可以触发事件"""
        return all(f in self.fragments for f in event.fragments_needed)

    def add_fragment(self, fragment: EventFragment):
        """添加碎片"""
        if fragment.fragment_id not in self.fragments:
            self.fragments.append(fragment.fragment_id)
```

### 2.2 与现有系统的整合

## 三、事件碎片系统

### 3.1 碎片获取

**探索行动**（替代原有调查）：
- 不消耗行动点
- 每次探索随机获得一个碎片
- 同一事件的碎片可能需要多次探索才能集齐

**碎片获取概率**：
- 基础概率：40%
- 知识值加成：知识÷20
- 灵感加成：灵感÷20

## 五、决策与能力检定

### 5.1 能力检定机制

```python
def check_ability(player: Player, ability_requirements: dict) -> bool:
    """能力检定

    基础成功率 = 50%
    每点能力值超过要求 +5%
    灵感加成 +2%

    Returns:
        是否成功
    """
    base_success = 0.5

    for ability, required in ability_requirements.items():
        player_value = player.skills.get(ability, 0)
        bonus = (player_value - required) * 0.05 if player_value >= required else 0

    # 灵感加成
    inspiration_bonus = player.inspiration * 0.02

    # 理智修正（理智低于50时，检定变得不稳定）
    sanity_modifier = 1.0
    if player.sanity < 50:
        sanity_modifier = 0.8  # 理智低下时更容易失败
    elif player.sanity < 30:
        sanity_modifier = 0.6  # 接近疯狂时大幅影响判断

    success_rate = (base_success + bonus + inspiration_bonus) * sanity_modifier

    return random.random() < success_rate
```

### 5.2 能力值影响调查深度

```python
def calculate_investigation_depth(player: Player, event: StoryEvent) -> dict:
    """计算调查深度

    根据玩家的能力值，决定能了解到多少信息
    """
    depth = {
        "surface": True,      # 表面信息
        "details": False,      # 细节信息
        "truth": False,       # 真相
    }

    # 检查各项能力
    total_bonus = 0
    for ability, value in player.skills.items():
        total_bonus += value

    if total_bonus >= 10:
        depth["details"] = True
    if total_bonus >= 25:
        depth["truth"] = True

    return depth
```

## 七、行动流程整合

### 7.1 探索行动

```
玩家选择"探索"行动：
    ↓
检查是否已有进行中的事件
    ↓
如果有 → 进入事件决策界面
    ↓
如果没有 → 进行探索
        ↓
    随机决定是否获得碎片
        ↓
    获得碎片 → 检查是否可以触发新事件
        ↓
    显示探索结果
```

## 十、待定问题

1. **事件进度是否需要存档？**
   - 建议：不保存，每次周目都是新的探索体验

2. **探索是否消耗行动点？**
   - 建议：不消耗，但有冷却时间（每周最多探索X次）

3. **碎片是否有有效期？**
   - 建议：无有效期，集齐后永久可触发

4. **多事件同时进行？**
   - 建议：一次只能进行一个事件
